# `@concierge-mantle/erc8004` — Domain Knowledge

## What this is

ERC-8004 ("Trustless Agents") is the canonical agent-identity + reputation primitive Concierge composes against. Every Concierge agent is an ERC-721 NFT in the Identity Registry; every successful tick writes a `giveFeedback` attestation to the Reputation Registry, producing a verifiable on-chain track record. This is the *verifiability* claim Concierge makes to judges (per ADR-004 — no zkML needed).

### ReputationRegistry (Feedback + Validation)

```solidity
// Submit feedback on an agent — Concierge calls this after every successful tick
function giveFeedback(
  uint256 agentId,
  int128 value,           // -100 to +100 (scaled by valueDecimals)
  uint8 valueDecimals,    // typically 2 → values are ×100
  string tag1,            // e.g. "tick-result"
  string tag2,            // e.g. "aave-v3-mantle/supply"
  string endpoint,        // e.g. "concierge.xyz/api/ticks/:tickId"
  string feedbackURI,     // off-chain JSON (IPFS) with reasoning + proof
  bytes32 feedbackHash    // hash of feedbackURI content
) external;

// Read aggregated reputation summary
function getSummary(
  uint256 agentId,
  address[] clients,      // empty array = all clients
  string tag1,            // empty string = any tag
  string tag2
) external view returns (uint64 count, int128 average, uint8 decimals);

// Read all individual feedback rows (paginated by clients)
function readAllFeedback(
  uint256 agentId,
  address[] clients,
  string tag1,
  string tag2,
  bool includeRevoked
) external view returns (
  address[] from,
  uint64[] indices,
  int128[] values,
  uint8[] valueDecimals,
  string[] endpoints,
  string[] feedbackURIs,
  bool[] revoked
);

// Read a single feedback entry
function readFeedback(uint256 agentId, address client, uint64 index)
  external view returns (int128, uint8, string, string, bool);

// Validation flow (response from agent to a feedback)
function appendResponse(
  uint256 agentId,
  address client,
  uint64 index,
  string responseURI,
  bytes32 responseHash
) external;

// Reads
function getResponseCount(uint256 agentId, address client, uint64 index, address[] clients) external view returns (...);
function getClients(uint256 agentId) external view returns (address[]);
function getLastIndex(uint256 agentId, address client) external view returns (uint64);
function getIdentityRegistry() external view returns (address); // links back to Identity

// Revoke a previously-submitted feedback (only by original giver)
function revokeFeedback(uint256 agentId, uint64 index) external;
```

## Integration pattern for Concierge

```typescript
// packages/concierge/erc8004/src/provider.ts
import { createPublicClient, createWalletClient, http } from 'viem';
import { mantle } from 'viem/chains';
import { identityRegistryAbi, reputationRegistryAbi } from './abis';

export const ERC8004_MANTLE = {
  identity: '0x8004A169FB4a3325136EB29fA0ceB6D2e539a432',
  reputation: '0x8004BAa17C55a88189AE136b182e5fdA19dE9b63',
} as const;

export function createErc8004Provider(opts: { rpcUrl: string }) {
  const publicClient = createPublicClient({ chain: mantle, transport: http(opts.rpcUrl) });

  return {
    // Action: register a new agent (mint NFT)
    register: {
      description: 'Mint a new Concierge agent identity NFT on ERC-8004',
      inputSchema: z.object({
        tokenURI: z.string().url(), // IPFS or HTTPS URL to agent manifest
        wallet: z.string().regex(/^0x[a-fA-F0-9]{40}$/),
      }),
      execute: async (input, ctx) => {
        const hash = await ctx.walletClient.writeContract({
          address: ERC8004_MANTLE.identity,
          abi: identityRegistryAbi,
          functionName: 'register',
          args: [input.tokenURI],
        });
        return { txHash: hash, action: 'register', registry: 'identity' };
      },
    },

    // Action: attest a tick result (called automatically by tick loop after execute())
    attestTick: {
      description: 'Write an ERC-8004 reputation feedback for a completed tick',
      inputSchema: z.object({
        agentId: z.bigint(),
        value: z.number().int().min(-10000).max(10000), // -100.00% to +100.00%
        tag1: z.string().max(64),
        tag2: z.string().max(64),
        endpoint: z.string().url(),
        feedbackURI: z.string().url(),
        feedbackHash: z.string().regex(/^0x[a-fA-F0-9]{64}$/),
      }),
      execute: async (input, ctx) => {
        const hash = await ctx.walletClient.writeContract({
          address: ERC8004_MANTLE.reputation,
          abi: reputationRegistryAbi,
          functionName: 'giveFeedback',
          args: [
            input.agentId,
            input.value,
            2, // valueDecimals
            input.tag1,
            input.tag2,
            input.endpoint,
            input.feedbackURI,
            input.feedbackHash,
          ],
        });
        return { txHash: hash, action: 'attestTick' };
      },
    },

    // Read: agent's aggregated reputation
    readReputation: async (agentId: bigint) => {
      return await publicClient.readContract({
        address: ERC8004_MANTLE.reputation,
        abi: reputationRegistryAbi,
        functionName: 'getSummary',
        args: [agentId, [] /* all clients */, '' /* any tag1 */, '' /* any tag2 */],
      });
    },
  };
}
```

## Mechanics / mental model

**Identity model:** every Concierge user gets ONE agent NFT minted on registration. The NFT's `tokenURI` points to an IPFS-hosted JSON manifest describing the agent (name, owner, goal categories, capabilities). The NFT owner is the user's wallet (or smart account); `setAgentWallet` binds an additional execution wallet (the session-key kernel) to the agent.

**Reputation model:** every successful tick triggers a `giveFeedback` write. The `value` is a signed int128 with `valueDecimals=2`, so `+150` means +1.50 (a +1.5% successful outcome attestation). The `tag1/tag2` pair lets us filter feedback by category (e.g., `tag1="tick-result"`, `tag2="aave-v3-mantle/supply"`). The `feedbackURI` points to an IPFS JSON with full reasoning + tx hash + before/after snapshot.

**Aggregation:** `getSummary` returns count + weighted average across requested tags, which is what the `/agent/:id` reputation page renders. The user (and judges) can drill into individual feedback entries via `readAllFeedback`.

**Response flow:** if an external party (e.g., another protocol or auditor) wants to challenge a feedback, they can call `appendResponse`. Concierge v1 doesn't implement the validation flow; it's documented for awareness.

**Off-chain content addressing:** `feedbackURI` + `feedbackHash` is the standard IPFS pattern — URI is the gateway URL (or `ipfs://` URI), hash is the keccak256 of the canonical content. Concierge stores the JSON in IPFS via Pinata or web3.storage (decide at integration); the hash is computed locally before write.

## Risks + edge cases to handle in specs

1. **Identity minting requires gas.** First-time users pay for the `register()` tx. Mitigation: agent runs cost is borne by Concierge during onboarding (sponsored via Pimlico paymaster) — pass the user's wallet as `setAgentWallet` post-mint.
2. **`setAgentWallet` requires EIP-712 signature.** This is a non-trivial UX step. Mitigation: pre-compute the typed-data hash on the frontend, prompt the user once during onboarding.
3. **Reputation write costs gas per tick.** ~50K gas per `giveFeedback` write. At Mantle gas costs (~$0.001/tx) this is negligible, but aggregate over 100s of ticks/day per user → ~$3/month worst case. Acceptable.
4. **`feedbackURI` content rot.** If IPFS pin drops, the off-chain reasoning is lost. Mitigation: dual-pin to Pinata + web3.storage; hash check on read to detect tampering.
5. **Revocation by giver only.** If Concierge erroneously attests a bad tick, only Concierge (the giver) can revoke. Document this in the user flow — give the user a way to flag attestations they think are wrong, which Concierge investigates and revokes manually.
6. **Agent NFT is transferable.** Standard ERC-721 transfer works — the user could sell their Concierge agent (with its reputation track record) on a secondary market. Treat as feature, not bug.

## Open questions for spec writer

1. **IPFS hosting:** Pinata vs web3.storage vs both? (Recommendation: both, for redundancy.)
2. **Agent manifest JSON schema:** what fields go in the `tokenURI` JSON? (Suggest: name, ownerWallet, createdAt, capabilities[], maxGoal $, version.)
3. **Reputation value scale:** how do we map a tick outcome to a `value` int128? (Suggest: +100 = profitable execution, +50 = neutral safe execution, -100 = loss-causing execution, scaled by % impact.)
4. **Validation flow:** ship v1 with `appendResponse` not implemented, or wire a stub? (Suggest: stub for v1, full implementation v1.1.)
5. **Cross-chain reputation:** v1.1+ should we read reputation from Ethereum Mainnet too (since the same agentId is portable)? (Suggest: yes, document the cross-chain read path.)
