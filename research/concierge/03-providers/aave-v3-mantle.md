# Aave V3 Mantle ‚Äî Concierge Domain Knowledge

## What this is
Aave V3 is the on-chain money market Concierge uses to (a) earn supply APY on idle stablecoins via `aTokens`, (b) borrow USDC/USDT0 against sUSDe/USDe/USDC collateral when the user's goal requires rebalancing or position management, (c) read health-factor + reserve data every tick for the agent's risk-management decisions. It is the **single most load-bearing action provider** for Concierge ‚Äî Aave's reserve data drives the agent's plan/simulate phases, and Aave's E-Mode 1 unlocks sUSDe-as-collateral capital efficiency.

Concierge uses **E-Mode Category 1 ("sUSDe Stablecoins", LTV 90.00% / LT 92.00% / Bonus 4%)** whenever sUSDe is supplied as collateral ‚Äî without `setUserEMode(1)`, sUSDe has LTV 0 in general mode (silent-failure trap). Captured below.

## Key functions / ABI surface

All functions live on `Pool` (`0x458F293454fE0d67EC0655f3672301301DD51422`). Use `viem`'s `getContract({abi: poolAbi, address})`.

```solidity
// Deposit collateral; mints aToken to `onBehalfOf`.
function supply(address asset, uint256 amount, address onBehalfOf, uint16 referralCode);

// Withdraw collateral. Pass `type(uint256).max` to withdraw all.
// REVERTS if HF would drop below 1.
function withdraw(address asset, uint256 amount, address to) returns (uint256);

// Borrow. interestRateMode: 2 = variable (the only mode supported on V3).
// referralCode = 0 always on V3.
// REVERTS with "COLLATERAL_CANNOT_COVER_NEW_BORROW" if HF < 1 post-borrow.
function borrow(address asset, uint256 amount, uint256 interestRateMode, uint16 referralCode, address onBehalfOf);

// Repay debt. Pass `type(uint256).max` to repay all (will pull from msg.sender).
function repay(address asset, uint256 amount, uint256 interestRateMode, address onBehalfOf) returns (uint256);

// Toggle E-Mode for the user's whole position.
// 0 = no E-Mode, 1 = sUSDe, 2 = USDe.
function setUserEMode(uint8 categoryId);

// Critical for HF telemetry ‚Äî Concierge reads this every tick.
function getUserAccountData(address user) returns (
    uint256 totalCollateralBase,    // USD * 1e8
    uint256 totalDebtBase,
    uint256 availableBorrowsBase,
    uint256 currentLiquidationThreshold, // bps; e.g. 9200
    uint256 ltv,                         // bps; e.g. 9000
    uint256 healthFactor                  // 1e18 scaled. HF=1.5 -> 1.5e18
);

// Switch a single asset between collateral-on and collateral-off.
function setUserUseReserveAsCollateral(address asset, bool useAsCollateral);
```

### Gotchas
- `referralCode` must be `0` (V3 ignores it but reverts on non-whitelisted refs in some forks ‚Äî pass 0 always).
- `borrow` requires the asset to be enabled for borrowing on Mantle deployment; check `getConfiguration(asset)` bitmask via the DataProvider.
- `withdraw(asset, max, to)` only withdraws the user's aToken balance, but the call still reverts if HF would drop. Concierge must pre-compute "max safe withdraw" using `getUserAccountData()`.
- aTokens **are** the receipts ‚Äî Concierge holds them in the smart account; no separate ledger.
- After `supply()`, `setUserUseReserveAsCollateral(asset, true)` is the default but Concierge should explicitly re-assert it (some assets are configured as collateral-disabled by governance after listing).
- `setUserEMode(1)` will revert if any of the user's existing borrows aren't in the E-Mode category's borrowable list. Order: enter E-Mode **before** the first borrow that requires the higher LTV.

## Integration pattern for Concierge

### Package: `@mpilot/aave-v3-mantle`
Exports:
- `actions.supply({asset, amount})`
- `actions.borrow({asset, amount})`
- `actions.repay({asset, amount})`
- `actions.withdraw({asset, amount})`
- `actions.setEMode({categoryId})`
- `selectors.getUserAccountData(user) ‚Üí {hf, ltv, collateralUSD, debtUSD}`
- `selectors.maxSafeBorrow({user, asset, targetHF})` ‚Äî pure compute helper

### Plan ‚Üí Simulate ‚Üí Propose ‚Üí Execute
1. **Plan**: Concierge tick computes the desired action (e.g. "borrow 1000 USDC against sUSDe"). Pulls live oracle prices via `AaveOracle.getAssetsPrices(address[])` to compute post-action HF.
2. **Simulate**: `eth_call` the encoded `borrow()` with `from = smartAccount` against a forked Mantle RPC (or via Tenderly bundle simulation). Reject if reverts.
3. **Propose**: Build the UserOperation calldata. Include a pre-borrow `setUserEMode(1)` if not already in E-Mode. Sign with session key (ERC-4337) scoped only to `Pool` + the 5 functions above.
4. **Execute**: Send to Mantle's ERC-4337 bundler. On success, read the receipt's `Borrow` event for the actual amount (in case of dust rounding).

### Error handling
| Aave error | Concierge action |
|---|---|
| `COLLATERAL_CANNOT_COVER_NEW_BORROW` | Reject plan in simulation; lower borrow amount; alert if user spend exceeds borrow capacity |
| `HEALTH_FACTOR_LOWER_THAN_LIQUIDATION_THRESHOLD` | Refuse withdraw; require partial repay first |
| `PRICE_ORACLE_SENTINEL_CHECK_FAILED` | Pause tick; oracle stale (Mantle oracle uses Chainlink + Redstone); Concierge waits 1 block |
| `RESERVE_FROZEN` / `RESERVE_PAUSED` | Mark provider degraded; pause all actions on that asset; fail open to user notification |
| Generic revert without reason | Bubble up; emit attestation with `"status": "failed"` and the calldata |

## Mechanics / mental model

### Health Factor
```
HF = (Œ£ collateralUSD_i √ó LT_i) / (Œ£ debtUSD_j)
```
- HF ‚â• 1 ‚Üí safe. HF < 1 ‚Üí liquidatable.
- E-Mode 1: LT for sUSDe = 9200 (92.00%). LTV = 9000.
- In E-Mode 1, only **sUSDe Stablecoins category** assets count for borrowing (USDC/USDT0/sUSDe/USDe).
- Concierge target HF = **1.6** at borrow, hard floor = **1.25** triggers auto-deleverage tick.

### Interest rate model
Aave V3 uses a two-slope IRM: `optimalUtilization` (e.g. 90% for stablecoins), `slope1` (rate up to optimal), `slope2` (steep slope above optimal). Variable borrow APR moves block-by-block. Concierge does NOT need to model this ‚Äî just call `getUserAccountData()` for current totals and `getReserveData(asset)` for live `liquidityRate` (supply APR) and `variableBorrowRate` (borrow APR) ‚Äî both `ray` (1e27) per second.

### Reserve config bitmask
`PoolDataProvider.getReserveConfigurationData(asset)` returns LTV, LT, liquidationBonus, decimals, isActive, isFrozen, borrowingEnabled, stableBorrowEnabled (false on V3), isPaused. Concierge reads on bootstrap + every 100 ticks.

### Liquidation cascade
At HF < 1, anyone can call `liquidationCall(collateralAsset, debtAsset, user, debtToCover, receiveAToken)` and seize up to 50% of debt (CLOSE_FACTOR=0.5) for `debtToCover * (1 + bonus)` collateral. E-Mode 1 bonus = 4% ‚Üí liquidator profit. Concierge's **prime directive**: never let HF approach the threshold; if approached, repay-then-deleverage.

## Risks + edge cases

1. **Oracle latency**: Mantle's AaveOracle wraps Chainlink + Redstone. If the underlying feed staleness check trips, borrow/withdraw revert. Concierge: catch + retry on next tick (‚â§ 2 retries before pausing).
2. **sUSDe price drift**: sUSDe's USD price ‚âà 1.23 (rising ‚Äî it's accruing yield). Concierge must compute HF using the **oracle's reported price**, not assume $1. Mistakes here = phantom liquidations.
3. **E-Mode borrow restriction**: In E-Mode 1, you cannot borrow assets outside the category (e.g. no WETH borrow). Concierge enforces at planner level.
4. **Borrow cap / supply cap**: each reserve has caps. Hitting them reverts. Concierge polls `getReserveCaps(asset)` and treats >95% utilization as "asset throttled".
5. **Isolation mode**: some assets (rare) are isolation-mode-only ‚Äî they pair with debt ceilings. Currently no Mantle reserve has this active, but Concierge must check `eModeCategory + borrowingEnabled + isolation` flags on each tick rather than caching forever.
6. **Aave pause**: Mantle deployment ACL admin (`0x70884634D0098782592111A2A6B8d223be31CB7b`) can pause the protocol globally. Concierge's `provider.health()` must check `Pool.paused()` (inherited from Pausable on V3.1).
